Q7Rate limit on Portfolio token — is the 200 req / 30 sec limit per token or per property when using a Portfolio Access Token?
What this means in plain terms: Mews allows 200 API calls every 30 seconds per access token — think of it as a flow limiter on a tap. With a standard per-property setup, each property gets its own token and its own 200-call budget. With a Portfolio token (one token covering all properties), the question is: does that 200-call budget apply to the whole portfolio combined, or does each property still get its own 200-call window through the single token?

Why it matters for Airnest: We currently have 4 properties (Bandama, Veintiuno, Masanet, Emete). Our nightly sync pulls reservation, billing, and payment data for each. If the limit is shared across the portfolio, a burst of 4 properties × ~15 calls each = 60 calls, well within budget. But as the portfolio grows toward 15–20 properties, a shared 200-call limit could become a bottleneck if all properties sync at the same time. Knowing this now lets us design the sync scheduler correctly — either staggering properties to spread load, or batching calls more aggressively.
Q8Webhook retry behaviour — what is Mews' retry policy if our webhook endpoint is briefly unavailable? Is there a dead-letter queue or manual replay option?
What webhooks are: When something changes in Mews — a reservation is created, a payment arrives — Mews instantly sends an automatic HTTP notification (a "webhook") to our server URL. Our server needs to be online and respond within a few seconds to acknowledge receipt.

The risk: If our server is briefly down — restarting after a software update, recovering from a blip — Mews will try to send the notification to a URL that isn't responding. The question is: does Mews try again automatically (and for how long)? Or is that notification permanently lost?

Why the answer changes what we build: If Mews retries for several hours, we can tolerate brief outages — our nightly 03:00 sync also catches up on any missed data. But if Mews drops notifications after one failed attempt, we need a more robust server setup (potentially a message queue like Redis that buffers incoming notifications even if our main process is momentarily busy). The answer tells us exactly how much resilience we need to build into the infrastructure from day one.

Q4What is your expected API call frequency?
Nightly bulk sync at 03:00 per property — pulls reservations/getAll, bills/getAll, payments/getAll, customers/getAll with date filters. Intra-day updates via webhooks (no polling). Live calls only for current-day data and the direct booking creation flow. Respects 90-day chunking for reservation intervals.
Q5How do you handle guest PII?
Encrypted, access-controlled, time-limited, and never shared.

What PII means: PII stands for Personally Identifiable Information — guest name, email address, phone number, nationality, passport or ID number. Essentially any data that can identify a specific person. As a Spanish company operating under European law, Airnest is subject to GDPR, which requires this data to be stored securely, used only for stated purposes, and not kept longer than necessary.

How we handle it in practice:
• Encrypted storage — guest data in the Ops Brain database is encrypted at rest. This means even if someone physically accessed the server, they could not read the guest data without the encryption key.
• Credential isolation — API tokens and passwords live in .env configuration files that are never uploaded to GitHub. They exist only on the server and on authorised developer machines.
• 24-month auto-delete — guest records are automatically removed 24 months after the guest checks out. This matches the standard Spanish hospitality compliance window (Modelo 179 reporting obligations).
• No third-party sharing — guest data is not passed to any external service beyond the systems that originally provided it (Mews). We do not send guest names, emails, or IDs to marketing platforms or analytics tools.
• Least privilege per credential — each API token is scoped to only what it needs. The Mews connector can read reservations and create guest profiles, but it cannot delete records, charge credit cards, or access other integrations' data.
Q6Why both ResourceUpdated webhook and Resource WebSocket event?
Different delivery guarantees. Webhooks (2–5 min latency) for guaranteed delivery and audit trail. WebSocket for lower-latency dashboard refresh on housekeeping signals where near-real-time matters. They serve complementary purposes, not duplicate ones.

Q10Are you respecting Mews' rate limits?
Yes — the connector is designed from the ground up to stay well within the 200 req / 30 sec limit.

What rate limits mean: Mews allows up to 200 API calls within any 30-second window. Exceed this and Mews temporarily blocks further calls until the window resets — similar to a data tap with a built-in flow limiter. Staying within limits is essential: a connector that trips the rate limit will start failing calls and missing data mid-sync.

How we stay within limits:
• 90-day reservation chunks — Mews rejects any reservation query spanning more than 100 days. A full-year data pull = 5 chunked calls (each covering 90 days), not one oversized request that would be rejected.
• Cursor-based pagination — when Mews has more data than fits in one response, we follow Mews' standard pagination pattern, requesting each page in sequence.
• TTL caching — room inventory and rate plans are cached for 1 hour. Reservations and availability are cached for 5 minutes. The vast majority of dashboard requests are served from cache, not from live API calls.
• Staggered nightly sync — bulk data pulls run at 03:00 when Mews activity is low. Properties sync sequentially, not in parallel.
• Webhook-driven intra-day — between nightly syncs, real-time updates come via Mews push notifications (webhooks), not by the connector polling the API repeatedly. Zero unnecessary API calls during business hours.
Q11What language and runtime are you using?
Python on a private Hetzner cloud server — industry-standard, GDPR-compliant, low-cost.

• Python — widely used for API integrations and automation. Airnest already uses it for financial scripts and the Guesty→Mews migration pipeline.
• httpx (async) — the library that makes the actual HTTP calls to the Mews API. Async means the connector doesn't freeze while waiting for Mews to respond — it can handle multiple operations efficiently.
• FastAPI — a lightweight Python web framework that receives the webhook POST messages from Mews. Fast, well-documented, industry-standard choice for small API receivers.
• Hetzner VPS (CX22, ~€4/month) — a German cloud server provider. GDPR-compliant with EU data centres. Significantly cheaper than AWS or Google Cloud and well-suited to low-traffic internal services.
• SQLite — a simple, file-based database. No separate database server to manage. Very low maintenance and appropriate for Airnest's data volume.
• .env credential files — API tokens and passwords are stored in configuration files explicitly excluded from version control (gitignored). They exist only on the server and authorised developer machines.
• Direct Mews token handling — no third-party library sits between our code and the Mews API. We inject ClientToken and AccessToken directly into each request body ourselves, exactly as Mews' documentation specifies.
Q12What happens if the webhook endpoint goes down?
Nightly sync provides the safety net. Even if webhooks are missed, the 03:00 bulk sync will catch up on any deltas from the previous day. VPS uptime will be monitored. We'll ask Milan about Mews' retry policy to assess whether a message queue buffer is needed.
